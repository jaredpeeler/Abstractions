Remarks

There are multiple different ways merchants communicate back information. Normally the user will fill out the form and hit a save button, they could also hit a cancel button. The merchant form will need to notify us of changes. Depending on the merchant this could be an event they trigger, a method they call, a page refresh, or a callback url.

This property needs to contain javascript to receive these messages, and communicate them back up to the DirectScale application. Essentially, the javascript needs to translates the merchant's messages into the format that DirectScale can consume, and pass them along.

The interface to communicate back to DirectScale is the following javascript functions:

  • DS_SavePaymentMethod(payment)Call this method to save the payment method. Payment is a json object. Example: 
    {
    token: "kyz92ngk03", // Provided by the merchant, used to reference the saved card. Required!
    last4: "4226", // Last 4 digits of card or account number, will be displayed to user when selecting their payment method.
    type: "Visa", // If one of the following, then that logo will be displayed next to the last4: "Visa", "MasterCard", "Amex", "Discover", "JCB".
    expireMonth = 3, // Expiration month of payment, defaults to 1 if not supplied.
    expireYear = 2022 // Expiration year of payment, defaults to 2099 if not suppplied. 
}
  • DS_AddPaymentError(message, data)Call if method if anything goes wrong. Be as specific as possible. The message and data will be shown to the user, and will be critical when trying to fix the issue. Data is any json object, and will be displayed as a string.
Examples

In this example, the merchant Iframe loads a callback url with a result of "SUCCESS", "FAILURE", or "CANCELED". The Javascript will watch for the iframe reloading (Ignoring the initial load) and handle the result. If the payment method is saved correctly, then a custom api is called to read the results (Don't call the merchant's API directly here from javascript because the required credentials would be publically visible and very insecure).

This example process flow is specific to this merchant and will likely be different for every merchant.

Note: some values in this example are dynamically populated when Head is set. Because Head is a string, c# string interpolation can be used to set specific values, ex: $"{HostedTransactionId}", and $"{AssociateId}" and others will need to be escaped: $"function () {{ }}".
